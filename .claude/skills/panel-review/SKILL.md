# 10人視点パネルレビュー

TOP BINSで大きめの設計判断をするときの定着した運用。アドリブでやると視点の抜けや馴れ合いが起きるため、この手順に従う。

10. 英語初級を対象にしたプロの英会話講師 — SLAの実務視点。「自己申告は盛られる」など教える側の経験則で施策の副作用を突く

ユーザーが「8人で」と言ったら1〜8のみ、「10人で」なら全員。

## 手順

### 1. 事実を先に取る（意見より前）
議題に関係する**実データを必ずコードや計測から取ってから**議論を始める。パネルの价值は「もっともらしい一般論」ではなく、実態に根ざした指摘にある。例: 「命令文が多すぎる」議題なら、まず`ROLEPLAY_SCENES`を実際に集計して「175件中80件(46%)が命令・宣言文、カテゴリ偏りはpitch24/manager20…」まで出す。

### 2. 各自の独立した指摘
各メンバーが**自分の専門領域から**見た指摘・提案を出す。全員一致の予定調和にしない — 特に:
- ペルソナ2人は具体的な体験談・言葉づかいで語らせる（「"簡単な会話ができる"の意味がそもそもわからない」のような一人称の詰まり）
- 過去のプロジェクト決定（design-log参照）と矛盾する提案が出たら、その矛盾自体を議論の材料にする

### 3. 対立軸を明示する
賛成の羅列ではなく、**意見がぶつかっている軸**を1〜3個明示する（例: 「足場を厚くする vs 高速オンボーディングを守る」）。対立が無い議題はそもそもパネルにかける価値が薄い。

### 4. 収束
会議として結論を出す。出力形式:

```
## 主な意見の対立軸
（軸ごとに、誰が何を主張し、どうトレードオフしたか）

## 結論（Best N または採用案）
（各案: 何をするか / なぜ効くか / コスト / 採用しない場合のリスク）
```

ユーザーが「ベスト3」等の個数を指定したらそれに従う。指定が無ければ採用案1つ＋保留案を提示。

## 原則

- **新規コンテンツ量産に頼る案は一段疑う**: このプロジェクトはSeason量産を却下し「既存175シーンの再利用・足場かけの初期値変更」で解決してきた実績がある（docs/design-log.md参照）。
- 結論には必ず「実装コストの目安」と「既存設計（Day構造・entitled分岐・計測）への影響」を含める。
- パネルの結論はあくまで提案。実装に進む前にユーザーの承認を待つ。
