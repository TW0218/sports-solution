## PWA / Apple App 併存・マネタイズ

- **entitledで無料/Proを分ける機能の選定（2026-07-12、コスト試算の上で決定）**: 実際にAPIコストを消費する箇所（`api/coach.js`, `api/score.js`, `api/transcribe.js`）を洗い出した結果、MATCHDAY/ROLEPLAY Tier1(BRONZE)/ミッション類は1人1日あたり合計でも$0.01未満とごく軽微だが、`askAI()`（PITCH VOCABのAIコーチ、claude-sonnet系max1000トークン）は1回$0.005前後と突出して高く、かつ`api/*.js`は`_guard.js`のOrigin検査＋IPレート制限だけでユーザー単位の上限が一切ないことが判明した。これを踏まえMATCH_TICKET_MAX=1（旧3）、Mastery Rank Tier2+の生涯1回トライアル（`S.rpTrialUsed`、合否問わず挑戦した時点で消費）、AIコーチ1日3回（`AI_COACH_FREE_DAILY_MAX`）、Weak Foot Missionのentitled専用化を決定。
- **摩擦除去型プレミアム（Ek指摘への対応）**: コンテンツ制限型ではなく摩擦除去型（MATCH TICKET無制限・音声一括DL）にした。Day25で頭打ちになるコンテンツを課金で囲うと払った人ほど損をするため。PWAはentitled常時trueのためこの2機能は現状PWAでも常に無料（設計上合意済み。ネイティブ版でIAP実装した時点で初めて分岐として機能する）。
- **サーバ側フェアユースの意図的未実装**: entitled分岐は全てクライアント側のみ（`api/*.js`はユーザーIDを見ておらず、devtoolsから直接叩けば回避可能）。ユーザー単位のサーバ側クォータ（`api_usage`テーブル等）はIAP本体（Capacitorラップ後）と同時に実装する計画。
- **音声一括DLのファイル名事故**: `%`や`\`を含む既存ファイル名を生の文字列で`fetch()`すると400エラーになる実績があり、`encodeURIComponent()`必須のルールになった。

## 行動計測

- 外部アナリティクスSaaS不採用の理由: プライバシーポリシー未整備の段階で新しい第三者にデータを渡さないため。Supabaseなら既存の信頼境界内に収まる。
- **計測イベント一覧（2026-07-18時点のスナップショット、最新はコードの`track(`呼び出しを検索）**: `app_open`（ログイン成功時・セッション復元時）/ `day_complete`（`day`,`avg_score`）/ `matchday_played`（`score`,`verdict`,`is_review`）/ `mission_complete`（`mission`: pitch_quiz/locker_quiz/cards/shadowing）/ `make_it_yours_complete`・`make_it_yours_skip` / `streak_broken`（`previous_streak`。実際に途切れた時のみ、0→1は含めない）/ `scene_mastery_up`（`key`,`tier`,`hint`）/ `season1_complete`（`avg`。初回・周回問わず発火）/ `weak_foot_mission_start`（`cat`,`pending`）/ `audio_bulk_download_start`・`complete`（`count`,`failed`）/ `hint_used`（`key`,`tier`。初展開時のみ、開閉繰り返しで重複記録しない）/ `notification_sent`・`notification_clicked`（`variant`付き）。

## プッシュ通知

- 許可リクエストをDay1完了直後にした理由: 価値を体験する前に許可を求めると許可率が落ちるため（インストール直後には出さない）。
- 通知文言のA/Bテスト: `mdReviewQueue`の実データ（シーン名・件数・`fails`）で内容を組み立て、言い回しを2バリアントでテスト。送信時は`SUPABASE_SERVICE_ROLE_KEY`でRLSを迂回して`notification_sent`をinsert、クリック時はURLパラメータ`?notif=1&notif_variant=X&notif_exp=Y`経由で`notification_clicked`を記録し`history.replaceState`でURLから消す（`?reset=missions`と同じパターン）。バリアント別CTRは`supabase/analytics_queries.sql`のクエリ8番。

## ROLEPLAY / Mastery Rank

- **Tier2ヒントのタップ開閉化（2026-07-12、SLA専門家視点レビュー）**: 元々`scene.hint`を常時表示していたが「ヒントを見たか判定できない＝真の定着判定に使えない」という指摘でMATCHDAYの`mdToggleHint()`と同じパターンに統一。初展開時だけ`hint_used`を記録。
- **hintWord/hintWordJaの全件追加（2026-07-14）**: 選定基準は「`keys`（お手本回答）に実際に登場する現場語・感情語を1つ」（唯一の例外はdaily「賃貸の問い合わせ」のviewing＝NPC側の語を意図的に採用）。キー名を`en:`にしないのは`validate_audio.js`の誤検知回避（MIYチップの`eng:`と同じ理由）。
- **ヒント使用は昇格をブロックしない（2026-07-14、8人視点レビュー）**: Tier階段自体が足場の漸減装置（SILVER=指示文+単語→GOLD=単語のみ→TOP BINS=支援なし）なので、独立した習熟はTOP BINSまで登れば構造的に検証される。ヒント使用に罰を与えるとユーザーが「昇格を守るためにヒントを見ない」ようになり学習も計測も壊れる、という判断。代わりに①`scene_mastery_up`に`hint`フラグを載せ、②Tier2/3でヒント未使用のまま昇格した時だけ「NO HINT — 自力で決めた」の名誉表示を出す（Tier4はヒント自体が無いので表示しない）。**ルールを厳格化する場合は必ず`hint`フラグ付き昇格データのベースレートを見てから**（Tier2挑戦の大半がヒント使用なら、ゲート化はほぼ全員の昇格を止める）。
- **スピード判定の可変閾値**: 175シーンのkeysは3語〜9語まで幅があり固定秒数だと不公平なため、`基礎1.2秒+1語0.35秒`でAランク閾値、+2秒でBランク閾値（`rpSpeedThresholds()`）。TOP BINS（Tier4）だけはスコア合格に加えAランク相当のスピードも必須ゲート。
- **「25」の意味づけ（2026-07-12、ユーザーから「25は中途半端では」と指摘され検討）**: 30/31（1ヶ月）への変更は、①日付ゲートがなく一気に進められる設計上「1ヶ月毎日続けた」実体が伴わない、②175=7×25を崩すには新規シーン追加（Season量産却下と矛盾）か配分崩しが必要、の2点で見送り。代わりに**プレミアリーグのトップチームスカッド登録枠25人**にコピーだけ紐付けた（`renderRoleplayMenu()`と`renderSeasonComplete()`の2箇所）。
- **SEASON 1 COMPLETEハブ画面**: Day25完了は「ゴール」ではなく「まだ続く」を伝える設計。3CTA（弱点を潰す／次のSeasonへ／実戦MATCHDAY）。弱点データが無いユーザーには「弱点を潰す」を出さない。初回だけでなくDay25再訪のたび毎回出る（定常ハブ、一度きりの祝福画面にしない）。
- **Season 1〜4量産の却下**: ユーザーが8人視点で自主分析し「Season 1〜4を実装すべき」と提案 → クロスチェックの結果、新規シーン量産ではなくMastery Rank（既存シーンの難易度階層リプレイ）を採用。

## 初級学習者対応（置いてけぼり防止）

- **lineJa常時表示（2026-07-15）**: 中学2年生ペルソナを含む10人視点分析で「初級者はNPCの英語命令文自体を読めず、何を返せばいいか分からないまま離脱する」リスクが最有力と判定。`hint`は「どう答えるか」、`lineJa`は「NPCが何を言っているか」で役割が別。常時表示にした理由: セリフの意味は毎回必要な情報でヒントではない。配線は`renderRPChoose()`・`renderRPFreeResponse()`・`renderMdScene()`の3箇所。
- **keysJa＋選択肢音声プレビュー（2026-07-18）**: 「状況とNPCのセリフは分かるが、3択の英文が読めない」という残存リスクに対応。175件全部が"正解"の言い換え違いなので、3択は「間違い探し」ではなく「言い方を選ぶ」だけ→翻訳を出しても学習的デメリット無しと判断。既存`keyAudio`をプレビュー再生ボタン（`rpPlayChoice()`）に流用し新規音声0件で実現。選択後の音読ステップは変更なし（英語productionは必須のまま）。
- **S.selfLevel自己申告＋足場かけ（2026-07-18）**: 「選手とのオンライン英会話で、今のアプリは英語初心者にはスタンダードが高すぎると感じた」というユーザーの実体験が起点。Duolingo型の適性判定テストは「判定結果を投げ込む簡単な方のコンテンツが存在しないため着地点が無い」として見送り、1問だけの自己申告＋既存足場かけの初期値変更に留めた。①beginnerはTier2+ヒントをデフォルト展開、②MATCHDAY本番はROLEPLAY Day3まで待たせる（DEMOは対象外）、③Day1初回は単語カードで肩慣らし（`S.rpVocabWarmupDone`）。**自己申告の「盛り」補正**: Day1〜3でintermediateなのに平均40点未満ならbeginnerへ自動ダウングレード（`S.selfLevelAuto`で一度きり、本人に通知しない。スキップした人は対象外＝選択を尊重）。判定にヒント使用率を使わないのはDay1〜3がほぼBRONZE（ヒント無し）のため。
- **未着手の残り案**: 英国サッカー俗語（gaffer/lads/mate等）のタップ開閉グロッサリー。

## MATCHDAY

- **「相槌だけで通せてしまう」問題・第1弾（2026-07-12）**: 175件中80件（46%）の`line`が命令・宣言文で、AI採点は意図伝達しか見ないため"OK, got it."でも合格しやすい。8人視点レビューで4案（Chou=コンボ、Ek=ラリー拡張、Eyal=語彙可視化、von Ahn=事前ナッジ）→コンボ・ナッジ・語彙可視化を共通の語彙判定エンジン（`mdKeyVocab()`）に統合して採用。ラリー拡張はPro機能として保留。ナッジは録音停止後・採点前に1シーン1回のみ、ローカル判定でAPI費用ゼロ。VOCAB COMBOは「相槌だけの合格はコンボを切らないが伸ばさない」、チケットやスコアには絡めない（名誉表示のみ）。
- **第2弾: 命令文30件の疑問文入れ替え（2026-07-18）**: 第1弾実装後もなお「Yes/Got itで完結するシーンが多すぎる」というユーザー指摘。MATCHDAYの出題プールは`S.rpCompletedSceneKeys`からしか選ばれないため新規シーンの単純追加では解決せず、Day構造（175=7×25）も壊れる。→各カテゴリで最もYes/No的なシーンを疑問文ベース新規シーンと1:1置き換え（30件、ユーザーがxlsxで英文を手直しして確定）。新規音声120件生成。
- **弱点レポート＋Weak Foot Missions**: 新規データを持たず既存`S.mdReviewQueue`のカテゴリ別集計だけで実現。ホーム画面の弱点カードと`startWeakFootMission()`が同じ集計関数（`weakCategoryStats()`/`weakestCategory()`）を参照。

## MAKE IT YOURS

- **設計刷新の経緯（2026-07-12、ユーザーの複数回フィードバック）**: ①自由生成のハードルが高すぎる→例文追加 → ②例文とベースフレーズが同居して混乱 → ③「日本語の質問に答えて型の穴を埋める」設計へ全面刷新。旧`MIY_EXAMPLES`・旧`miyCheck()`/`miyShowResult()`（発話後AI採点）は廃止。
- **文法チェックは「採点」ではなく「変換」**: 発話前の変換ステップで完成文が確定するため、発話後の再チェックはしない。
- **スキップ救済**: `miySkip()`は変換済みならそのまま保存、下書きなら`saveDraftPhrase()`で変換して保存（`note`に「（下書き）」付き）。作文の努力を失わせない。

## UI事故と修正の詳細

- **sfx呼び出し漏れ（2026-07-12発見・修正）**: ドリル・ROLEPLAY・MATCHDAY・MAKE IT YOURSの4モードで正解/不正解の基本フィードバック音が抜けていた。高得点だけの派手な演出（`showCelebration()`等）を後から追加した際、基本音を引き継ぎ忘れたのが原因。
- **絵文字18箇所の一掃**: 目視レビューでは18箇所中9箇所しか見つけられなかったため、`npm run verify`に機械チェックを追加した。
- **MediaRecorder inactive デッドロック（2026-07-12発見・修正）**: `_stopDrillRec()`/`stopRPRec()`/`mdStopRec()`が同じ構造のバグを持っていた。「processing中」フラグを立ててから`state!=='inactive'`の時だけ`stop()`を呼ぶ設計だったため、iOSのオーディオセッション切替等でMediaRecorderが自然停止していると`onstop`が永久に発火せずフラグが立ちっぱなしで操作不能（「4回目のリトライでマイクがタップできなくなった」として発現）。修正: `inactive`ならフラグを立てずその場で失敗扱いの結果表示へ。描画開始時のフラグリセット安全網も追加。
- **`_transcribeAudio()`タイムアウト**: 元々fetchにタイムアウトがなく通信不調時に「認識中...」で無期限ハング。20秒のAbortControllerを追加（全録音モードが共通関数を使うため1箇所で全モードに効く）。
- **マイク赤固まり（2026-07-17発見・修正）**: `mdStartRec()`が`onclick`と背景色を書き換える設計で、`mdShowNudge()`/`mdNudgeRetry()`が元に戻していなかったためナッジ表示後にマイクが操作不能。RP/ドリルは`toggleXxxRec()`の単一エントリポイント設計（onclick固定）のため同種事故が起きない。
- **100dvh事故（2026-07-17発見・修正）**: ブラウザツールバー表示中の初期描画で`dvh`が「ツールバーが隠れた場合」想定の大きめの値になり、実際の可視範囲より内部高さが大きく計算され、MATCHDAYのHINTボックスが画面外に押し出されスクロールもできない状態に（overflowがCSS上発生しないためスクロールバーも出ない）。`100svh`（最小保証サイズ）に変更。
- **オンボーディングの文脈型ヒント（2026-07-12）**: Zhuo（UI/UX）の「3タブやモード間の繋がりが初見で分かりにくい」指摘に対し、大きなチュートリアルではなく各タブ初回描画時の1行バナー（`hintBanner()`/`S.seenHints`）で対応。高速な2ステップオンボーディングを崩さないため。
